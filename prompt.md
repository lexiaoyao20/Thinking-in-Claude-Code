# 优化后的 Prompt：《Claude Code 源码编程思想》

---

## 📌 使用说明

以下 Prompt 分为 **3 个阶段**，建议按顺序在 Claude Code 中执行。
每个阶段产出明确的交付物，下一阶段依赖上一阶段的输出。

---

## 阶段一：大纲设计（Outline Architect）

```
你是一位资深技术图书策划编辑，同时精通大型 CLI/AI Agent 系统架构。

## 任务
阅读 Claude Code 源码仓库（已提供），产出《Claude Code 源码编程思想》的完整书籍大纲。

## 写作对标
- 结构与深度对标 Bruce Eckel《Thinking in C++》——每章聚焦一个核心编程思想，从"为什么这样设计"出发，而非罗列 API。
- 每章包含：核心思想 → 源码剖析 → 设计权衡 → 读者迁移指南（如何应用到自己的项目）。

## 大纲要求

### 1. 章节覆盖维度（至少涵盖，可扩展）
- 整体架构哲学（分层、模块边界、依赖方向）
- 命令解析与调度机制（CLI → Command Router → Executor）
- Agent 循环与对话管理（消息流、上下文窗口、Tool Use 协议）
- 工具系统设计（Tool 注册、沙箱隔离、权限模型）
- 流式输出与终端渲染（Ink/React 终端 UI 架构）
- 错误处理与韧性设计（重试、降级、中断恢复）
- 配置与扩展机制（Settings层级、MCP 集成、Skills/Commands）
- 并发与性能（子进程管理、文件监听、缓存策略）
- 测试与工程质量（测试分层、Mock 策略、CI 流水线）
- 编程哲学总结（从 Claude Code 中可迁移的设计模式）

### 2. 每章大纲格式
```markdown
## 第 N 章：{章节标题}
- **核心思想**：一句话概括本章要传达的编程思想
- **关键源码模块**：列出涉及的核心文件/目录
- **子节**：
  - N.1 {子节标题} — {简述}
  - N.2 ...
- **费曼检验问题**：1-2 个问题，读者读完后应能用自己的话回答
- **Mermaid 图表规划**：列出本章计划使用的图表类型（流程图/架构图/时序图/状态机等）
```

### 3. 输出要求
- 产出一个 `OUTLINE.md` 文件
- 章节数量 12-18 章（含前言和附录）
- 每章预估篇幅标注（短/中/长）
- 章节之间标注依赖关系（哪些章需要先读）

## 约束
- 不要写正文内容，只输出大纲
- 优先从源码中提炼"思想"，而非描述"功能"
- 如果某个设计有明显的 trade-off，在大纲中标注
```

---

## 阶段二：任务分解与分工（Task Planner）

```
你是技术图书项目经理。基于已完成的 OUTLINE.md，执行以下工作：

## 任务
将大纲分解为可并行执行的写作任务列表，并设计多 Agent 协作方案。

## Agent 角色定义

| Agent 角色 | 职责 | 输出物 |
|-----------|------|--------|
| 🔍 Source Analyst | 阅读源码，提取关键设计模式、数据流、核心抽象 | 源码分析笔记（每章对应） |
| ✍️ Chapter Writer | 基于分析笔记撰写章节正文 | 单章 Markdown 文件 |
| 🧠 Feynman Reviewer | 用费曼学习法审查：能否用简单类比解释？有无知识诅咒？ | 审查意见 + 修改建议 |
| 📊 Diagram Designer | 设计并编写 Mermaid 图表，确保与正文配合 | 嵌入章节的 Mermaid 代码块 |
| 🔗 Consistency Editor | 跨章术语统一、交叉引用、避免重复 | 编辑修订记录 |

## 任务列表格式
```markdown
### Task-{序号}: {任务描述}
- **对应章节**: 第 N 章
- **执行 Agent**: {角色}
- **前置依赖**: Task-{X}, Task-{Y}（无则标注"无"）
- **输入**: {需要什么}
- **输出**: {产出什么文件}
- **验收标准**: {完成的判断条件}
```

## Brainstorm 环节要求
在生成任务列表前，先进行一轮 Brainstorm：
1. 每个章节最难写的部分是什么？为什么？
2. 哪些章节之间有强耦合，必须串行？
3. 哪些章节可以完全并行？
4. 读者最可能在哪里"卡住"？如何用费曼法化解？

将 Brainstorm 结论写入 `BRAINSTORM.md`，任务列表写入 `TASKLIST.md`。
```

---

## 阶段三：章节写作（Chapter Execution）

```
你是技术图书作者，同时扮演上述所有 Agent 角色，按 TASKLIST.md 逐步执行。

## 写作规范

### 风格要求
- 对标《Thinking in C++》：先讲"为什么需要"，再讲"怎么实现"，最后讲"如何迁移"
- 每个关键概念用【费曼学习法】三步呈现：
  1. 📖 用一段话向非程序员解释核心思想
  2. 🔬 展示对应源码片段并逐行剖析
  3. 🛠️ 给出读者可迁移的最小实践示例
- 行文口吻：专业但不学院派，像一位资深工程师在白板前讲解

### 源码引用规范
- 引用源码时标注文件路径和关键行号范围
- 大段源码（>30行）用折叠块包裹，配摘要说明
- 每段源码后必须有"设计意图解读"段落

### Mermaid 图表规范
- 每章至少 2 张 Mermaid 图表
- 图表类型建议：
  - 架构概览 → `graph TD` 或 `C4Context`
  - 请求/消息流 → `sequenceDiagram`
  - 状态变迁 → `stateDiagram-v2`
  - 模块依赖 → `graph LR`
  - 决策逻辑 → `flowchart`
- 图表前后必须有引导文字（不要裸放图表）

### 文件组织
```
book/
├── README.md           # 总目录（含各章链接和阅读指南）
├── 00-preface.md       # 前言
├── 01-architecture.md  # 第1章
├── 02-xxx.md           # 第2章
├── ...
├── appendix-a.md       # 附录A
└── glossary.md         # 术语表
```

### 每章 Markdown 模板
```markdown
# 第 N 章：{标题}

> **核心思想**：{一句话}

## N.1 为什么需要 {概念}？
{费曼式引入，用日常类比}

## N.2 Claude Code 如何实现
{源码剖析 + Mermaid 图表}

## N.3 设计权衡与替代方案
{为什么选 A 而非 B？trade-off 是什么？}

## N.4 迁移指南：应用到你的项目
{最小可行示例 + 注意事项}

## N.5 费曼检验
{如果你能回答以下问题，说明你真正理解了本章：}
- Q1: ...
- Q2: ...

## 本章小结
{3-5 个要点}
```

### 质量验收标准（每章必须满足）
- [ ] 核心思想清晰，非功能描述
- [ ] 至少引用 3 处关键源码并解读设计意图
- [ ] 至少 2 张 Mermaid 图表，且与正文呼应
- [ ] 包含费曼学习法的三步呈现
- [ ] 包含"迁移指南"——读者能带走的实践建议
- [ ] 无与其他章节重复的大段内容
- [ ] 术语与 glossary.md 一致

## 执行流程
对每一章，按以下顺序执行：
1. **Source Analyst**：阅读相关源码，输出分析笔记
2. **Brainstorm**：基于分析笔记，梳理本章叙事线
3. **Diagram Designer**：先画 Mermaid 图，确定可视化骨架
4. **Chapter Writer**：撰写正文，嵌入图表和源码
5. **Feynman Reviewer**：审查是否真正"讲明白了"
6. **Consistency Editor**：检查术语、交叉引用、与其他章节的衔接

每完成一章，输出到 `book/{filename}.md`。
全部完成后，生成 `book/README.md` 总目录。
```

---

## 💡 使用建议

1. **如果在 Claude Code 中使用**：将三个阶段的 Prompt 分步执行，每步确认产出后再进入下一步。
2. **如果在 Claude.ai 中使用**：可以一次性提供完整 Prompt，但建议先完成阶段一的大纲审核。
3. **源码输入**：确保 Claude Code 源码仓库已在工作目录中（或通过 `git clone` 获取）。
4. **迭代优化**：大纲和任务列表是活文档，写作过程中可以回溯调整。
